# D6: Stage 0 (Prerequisites), Stage 1 (Flick), Stage 2 (Lot)

## Context

Gate is a binary-only E2E test harness for the backlot workspace. Full design: `specs/GATE.md`. D1–D5 established the full infrastructure: types, assertions, reporting, subprocess execution, scratch directories, binary discovery, and stage runner. This deliverable fills in the first real stages.

## Dependencies

D1–D5 (all infrastructure). This deliverable replaces the stub implementations for Stage 0 (prerequisites), Stage 1 (flick), and Stage 2 (lot).

## What to Build

### Stage 0: Prerequisites (`gate/src/prereqs.rs`)

Runs automatically before any test stage. Not a test stage itself — blocks execution if environment is not ready.

```rust
/// Check all prerequisites. Returns Ok(()) or Err with a list of specific problems.
pub fn check_prerequisites(binaries: &BinaryPaths) -> Result<(), PrereqError>;

pub struct PrereqError {
    pub problems: Vec<String>,
}
```

Checks (in order):
1. **All required binaries exist** — already verified by binary discovery, but re-confirm.
2. **`~/.flick/providers` exists and is non-empty** — the directory must exist and contain at least one file.
3. **`~/.flick/models` exists and contains `fast`, `balanced`, `strong` aliases** — the directory must contain files named `fast.toml`, `balanced.toml`, `strong.toml`.
4. **`lot setup --check` passes** — run `lot setup --check` via the lot binary; exit 0 means sandbox is ready.

Each check that fails adds a specific message to `problems`. Examples:
- "~/.flick/providers directory is empty — run `flick provider add <name>` to register a provider"
- "~/.flick/models missing alias 'strong' — run `flick model add` to create it"
- "`lot setup --check` failed — run `lot setup` from an Administrator terminal"

Wire into `runner::run()`: call `check_prerequisites()` after binary discovery but before any stage. If it fails, print all problems and return exit code 2.

### Stage 1: Flick (`gate/src/stage/flick.rs`)

Replace the stub. Six tests exercising flick's CLI.

**Fixture files in `gate/fixtures/flick/`:**

Create minimal fixture files as needed. Flick is invoked via its CLI binary. Each test constructs the appropriate CLI args.

**Tests:**

1. **`basic-invocation`** — Call flick with a simple prompt using the `fast` model alias. Assert: exit 0, stdout parses as JSON, JSON has `status: "complete"`, non-empty `content`, valid `usage` block (has `input_tokens`, `output_tokens`).

2. **`chatcompletions-invocation`** — Same as above but using a Chat Completions provider model. If only one provider type is registered, this test may be a soft-fail or skip. For now, use the `balanced` model alias (likely a different provider). Assert same fields as basic-invocation.

3. **`tool-declaration-and-resume`** — First call: declare a tool (e.g., `get_weather` with a simple schema), provide a prompt that would trigger tool use. Assert: status is `tool_calls_pending`, `tool_calls` array non-empty. Second call: resume with tool results. Assert: status is `complete`, content references the tool result.

4. **`structured-output`** — Call with `output_schema` specifying a simple JSON schema (e.g., `{"type": "object", "properties": {"name": {"type": "string"}, "count": {"type": "integer"}}, "required": ["name", "count"]}`). Assert: exit 0, content parses as JSON, JSON has the required fields.

5. **`dry-run`** — Call with `--dry-run` flag. Assert: exit 0, non-empty stdout (request payload), no API call made (verify by checking that no `usage` block is present or that the output structure differs from a real call).

6. **`error-invalid-model`** — Call with model name `nonexistent-model-xyz`. Assert: exit code != 0.

**How to invoke flick CLI:** Read `flick --help` or the flick design doc to determine exact CLI arguments. The flick CLI takes input on stdin (JSON messages) and outputs JSON to stdout. Use `run_command()` from D4, piping input via stdin. If `run_command` doesn't support stdin piping, extend it — add an optional `stdin: Option<&str>` parameter.

**Note:** If `run_command` needs stdin support, add it in this deliverable. This is a natural extension — the function spawns a child, writes to its stdin, then waits for output.

### Stage 2: Lot (`gate/src/stage/lot.rs`)

Replace the stub. Eight tests exercising lot's CLI.

**Per-platform fixtures in `gate/fixtures/lot/`:**

Lot policies are YAML files. Create minimal policies for each test. Store as `fixtures/lot/policy_<name>.yaml` or `fixtures/lot/<platform>/policy_<name>.yaml`. The fixture loader picks the right one based on `cfg!(target_os)`.

**Tests:**

1. **`probe`** — Run `lot` with probe/capability detection args. Assert: exit 0, output mentions the platform backend (AppContainer/seccomp/Seatbelt).

2. **`setup-check`** — Run `lot setup --check`. Assert: exit 0.

3. **`fs-read-allowed`** — Create a file in the scratch dir. Run a sandboxed process (via lot CLI) with read access granted to the scratch dir. The sandboxed process reads the file and prints its content. Assert: exit 0, stdout contains the file content.

4. **`fs-write-allowed`** — Run a sandboxed process with write access granted to scratch dir. The process creates a file. Assert: exit 0, file exists after process exits.

5. **`fs-deny-overrides-read`** — Grant read to a parent dir, deny read to a specific file within it. Run sandboxed process that tries to read the denied file. Assert: non-zero exit (access denied).

6. **`network-denied`** — Run sandboxed process without network access. The process tries to make an HTTP request (e.g., curl or a simple TCP connect). Assert: non-zero exit or connection failure.

7. **`network-allowed`** — Run sandboxed process with network access. Try to reach a known endpoint. Assert: exit 0. **Mark as SoftFail** if it fails (infrastructure-dependent — network may be blocked by firewall).

8. **`timeout`** — Run a sandboxed process that sleeps forever, with a short timeout. Assert: process killed, exit code indicates timeout.

**How to invoke lot CLI:** Read `lot --help` or the lot design doc for exact args. Lot likely takes a policy file and a command to run in the sandbox.

**Scratch dir usage:** All file operations use the stage's scratch dir (`ctx.scratch_dir`). This is project-local under `target/gate-scratch/`, safe for AppContainer.

## TDD: Tests to Write First

### Unit tests (in source files)

For Stage 0:
1. `prereq_all_ok` — mock all checks passing → Ok
2. `prereq_missing_providers` — providers dir missing → Err with specific message
3. `prereq_missing_model_alias` — one alias missing → Err mentioning which alias
4. `prereq_multiple_problems` — multiple failures → Err listing all of them

For fixture loading:
5. `platform_fixture_selection` — verify the correct platform fixture path is returned on the current OS

### Integration tests (require real environment)

These are the stage tests themselves — they run against real binaries with real credentials. They are NOT `#[cfg(test)]` unit tests. They are the tests that `gate` runs. They are verified by running `gate --only flick` and `gate --only lot` manually.

The TDD approach here: write the test function bodies first (assertions and expected flow), then implement the CLI invocation logic to make them pass.
