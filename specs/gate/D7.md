## What to Build

### Stage 3: Reel (`gate/src/stage/reel.rs`)

Replace the stub. Five tests exercising reel's agent runtime CLI.

**Fixtures in `gate/fixtures/reel/`:**

Create workspace seed files for the agent to operate on:
- `fixtures/reel/workspace/` — a small directory with 2-3 known files (e.g., `hello.txt` with "Hello, World!", `data.json` with `{"key": "value"}`). These are copied into the scratch dir for each test.

Reel agent configs — check `reel --help` and the reel design doc for how to specify model, tools, grants, workspace root, and prompt via CLI args. The reel CLI likely takes a config file or CLI flags for these.

**Tests:**

1. **`readonly-session`** — Run reel agent with read-only grants (Glob + Read tools) on the seeded workspace. Prompt: "List all files and read hello.txt". Assert: exit 0, output contains "Hello, World!" (the file content), usage block present with non-zero tokens.

2. **`write-session`** — Run reel agent with write grants. Prompt: "Create a file called output.txt containing 'test output'". Assert: exit 0, `output.txt` exists in the workspace scratch dir, content matches expected.

3. **`nushell-execution`** — Run reel agent with NuShell tool enabled. Prompt: "Use NuShell to compute 2 + 2 and report the result". Assert: exit 0, session completes without crash. (The exact output is non-deterministic, but the process shouldn't fail.)

4. **`multi-turn`** — Run reel agent with a task requiring multiple tool calls. Prompt: "Read hello.txt, then read data.json, then create summary.txt with the contents of both files". Assert: exit 0, `summary.txt` exists, output indicates multiple tool calls were made (if reel reports tool call count, assert `tool_calls > 1`; otherwise just verify the multi-step outcome).

5. **`error-invalid-model`** — Run reel with model `nonexistent-model-xyz`. Assert: exit code != 0.

**Workspace setup:** Before each test (or before the stage), copy `fixtures/reel/workspace/` into `ctx.scratch_dir`. Each test operates on this copy.

### Stage 4: Vault (`gate/src/stage/vault.rs`)

Replace the stub. Five tests exercising vault's knowledge store CLI.

**Fixtures in `gate/fixtures/vault/`:**

Create a minimal vault config file:
- `fixtures/vault/config.yaml` — vault config pointing to the scratch dir as the store root. Check `vault --help` and the vault design doc for config format.

**Setup:** Each test run starts with a clean store. The stage setup function wipes and recreates the vault store directory in the scratch dir before running tests.

**Tests run in order** — vault tests are sequential and build on each other (bootstrap creates the store, record adds to it, query reads from it, reorganize restructures it).

1. **`bootstrap`** — Pipe requirements text into `vault bootstrap`. Assert: exit 0, `raw/` directory exists in store, `derived/` directory exists, changelog file exists, output contains usage information (tokens).

2. **`record-new`** — Run `vault record --mode new` with a document name and content. Assert: exit 0, new raw document file created.

3. **`record-append`** — Run `vault record --mode append` targeting the document from test 2, with additional content. Assert: exit 0, document file size increased (or content includes the appended text).

4. **`query`** — Run `vault query` with a question about the recorded content. Assert: exit 0, response is non-empty, response references or contains information from the recorded documents.

5. **`reorganize`** — Run `vault reorganize`. Assert: exit 0, derived docs may have changed, changelog has new entries (more lines than after bootstrap).

**Vault CLI invocation:** Check `vault --help` for exact args. The vault CLI likely takes a config file path and subcommands (bootstrap, record, query, reorganize).
